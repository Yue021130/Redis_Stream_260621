# Redis 高级数据结构：Bitmap 与 HyperLogLog 经典应用场景实战

## 1. 更新概述
除了经典的五大数据结构外，本次特别补充了 Redis 中专门为了应对**海量数据极其苛刻的内存限制**而诞生的两种高级数据结构：**Bitmap（位图）** 和 **HyperLogLog**。
它们的核心魅力在于：用极小极小的内存代价，完美解决上亿级用户规模的数据状态记录与基数统计问题。

## 2. 核心场景详细说明

### 2.1 Bitmap (位图) - 海量用户签到
- **原理**：Bitmap 本质上是 String 结构（最大容量 512MB），但它打破了按字节操作的限制，直接按 **bit (比特位)** 来操作。由于一个 bit 只有 0 或 1，极其适合表达 Boolean 类型的状态（如：是否签到、是否在线、是否点过赞）。
- **为什么海量签到不用 Set/Hash？** 
  假设有 1 亿注册用户，如果每天用 Set 存签到记录，一个月下来需要几个 GB 的内存。
  而使用 Bitmap，如果以 `userId` 作为偏移量（offset），1 亿个 bit 仅仅占用约 **12 MB** 的内存。如果以 `月份` 作为键，用户的整月 31 天签到记录只需要 31 个 bit，即使用户量巨大，每个用户的签到开销也仅仅是 **4 字节**不到！
- **操作指令**：
  - `SETBIT key offset value`：设置某天（offset）的状态（1 为已签到）。
  - `GETBIT key offset`：查询某天是否签到。
  - `BITCOUNT key`：极速统计比特数组中 1 的个数，即完美计算当月累计签到天数。

### 2.2 HyperLogLog - 亿级 UV (独立访客) 统计
- **原理**：基于数学概率论的基数（不重复元素个数）估计算法。标准误差率为极其微小的 **0.81%**。
- **为什么要用它？** 
  在前面的 Set 案例中，我们演示了用 Set 存 UV 访问量（通过 `SADD` 和 `SCARD`）。在几万十万级别的访问量下没问题。但如果淘宝双十一某爆款页面的访问独立用户达到了 **1 亿**，如果用 Set 存储这 1 亿个完整的 String 类型的 UserID，单单这一个 key 就会消耗近 **1 GB** 内存。如果有成千上万个页面，Redis 瞬间内存撑爆。
  而奇迹般的 **HyperLogLog 无论塞进去多少个千万级、亿级的元素，一个 Key 最大只占用 12 KB 的内存**！
- **适用场景**：只关心去重后的总数量，不关心具体是哪些人访问了，且能容忍极小误差的超级海量计数场景。
- **操作指令**：
  - `PFADD key element`：添加访问过的用户 ID。
  - `PFCOUNT key`：获取该页面的 UV 估算总数。

## 3. 测试接口指南 (入口路径 `/api/redis-usecases/advanced`)

**Bitmap 签到测试：**
- **点击签到**：`POST /bitmap/sign/{userId}` *(连续请求同一天只会提示已签到过)*
- **查询今日是否签到**：`GET /bitmap/sign/{userId}/check`
- **查询当月总计签到天数**：`GET /bitmap/sign/{userId}/count`

**HyperLogLog 亿级 UV 测试：**
- **增加用户访问记录**：`POST /hll/uv/{pageId}/visit?userId=user999` *(可以使用脚本并发传递成千上万个不同的 userId 进去)*
- **获取 UV 估算值**：`GET /hll/uv/{pageId}/count`
